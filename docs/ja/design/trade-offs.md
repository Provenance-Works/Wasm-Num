# トレードオフ

> **対象読者**: 開発者、アーキテクト、コントリビューター

wasm-num の設計における主要なトレードオフと、検討した代替案について記録します。

## 1. BitVec 統一 vs. 個別型

**選択**: すべての Wasm 数値型を `BitVec N` エイリアスとして統一表現

**メリット**:
- reinterpret がゼロコスト（アイデンティティ関数）
- 演算間の変換オーバーヘッドなし
- `BitVec` に対する Mathlib の補題を直接活用可能

**デメリット**:
- `I32` と `F32` は同一型（`BitVec 32`）のため型安全性が低い
- 意味的に異なる型が型レベルで区別されない

**結論**: 形式化においてはビットレベルの汎用性が型安全性よりも重要です。WebAssembly 仕様自体がビットパターンを型間で共有しています。

## 2. Set α vs. モナド

**選択**: 非決定性に `Set α`（`α → Prop`）を使用

**メリット**:
- シンプルで数学的に自然
- メンバーシップ（`∈`）と集合演算による直接的な証明
- モナドのインフラストラクチャが不要

**デメリット**:
- 非決定的関数の合成に明示的な集合内包が必要
- 直接実行不可（決定的ラッパーが必要）

**検討した代替案**: `NondetM α = List α` モナド。このユースケースには過剰で、`Set α` のほうが Lean の型理論で自然です。

## 3. 型クラス vs. モジュールパラメータ

**選択**: 外部境界に型クラス（`WasmFloat`、`Profile`）を使用

**メリット**:
- Lean の暗黙的インスタンス解決と自然に統合
- `instance` 宣言でグローバル / ローカルの制御が可能

**デメリット**:
- 複数のインスタンスが存在する場合にインスタンスの曖昧性が生じうる
- `variable [WasmFloat N]` がすべてのシグネチャに伝播

**検討した代替案**: `structure WasmFloatImpl` を明示的にパラメータとして渡す方式。ボイラープレートが多く、Lean の型クラスメカニズムを活用しない非慣用的なアプローチになります。

## 4. 純粋 Lean vs. C FFI

**選択**: C FFI を一切使用しない純粋 Lean

**メリット**:
- 完全に検証可能な信頼されたコンピューティングベース
- ネイティブビルド依存関係なし
- すべてのプラットフォームで同一の動作

**デメリット**:
- ハードウェア浮動小数点数の速度を活用不可
- デフォルトの float スタブはプレースホルダ値を返す

**検討した代替案**: Berkeley SoftFloat の C FFI リンク。検証ギャップとネイティブビルド依存関係のため却下。型クラスアプローチにより、ユーザーは外部でブリッジを提供可能です。

## 5. 証明の分離 vs. コロケーション

**選択**: 定義と証明を厳密に別ファイルに分離

**メリット**:
- `lake build WasmNum` がプルーフチェック不要で高速
- 定義ファイルがクリーンで読みやすい
- プルーフの失敗が定義の開発をブロックしない

**デメリット**:
- 並行するディレクトリ構造の保守が必要
- 証明を追加せずに定義を追加してしまう可能性

**検討した代替案**: 同一ファイルにコロケーション。開発時の反復におけるビルド速度のペナルティが許容できないため却下。

## 6. ByteArray vs. Fin インデックスベクター

**選択**: `FlatMemory` のデータ表現に `ByteArray` を使用

**メリット**:
- Lean のネイティブ `ByteArray` を直接活用（効率的）
- バイト単位のアクセスが自然
- WebAssembly のメモリモデル（リニアバイト配列）と直接対応

**デメリット**:
- 境界チェック証明が必要（`data.size = pageCount * pageSize`）
- Fin インデックスのような静的保証がない

**検討した代替案**: `Vector (UInt8) size` や `Fin n → UInt8`。`ByteArray` のほうが実用的で、WebAssembly 仕様のバイトレベルメモリモデルと直接対応しています。
