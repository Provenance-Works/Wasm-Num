# ADR-0002: BitVec N を統一表現として採用

| | |
|---|---|
| **ステータス** | 承認済 |
| **日付** | 2025 |
| **決定者** | wasm-num メンテナー |

## コンテキスト

WebAssembly には4つの数値型と1つのベクター型があります：
- `i32` — 32ビット整数
- `i64` — 64ビット整数
- `f32` — 32ビット浮動小数点数
- `f64` — 64ビット浮動小数点数
- `v128` — 128ビット SIMD ベクター

これらの型はビット表現を共有しています：`f32` と `i32` はどちらも32ビットであり、`reinterpret` でゼロコスト変換されます。SIMD レーンは同じ128ビットをさまざまな幅で解釈します。

## 決定

すべての WebAssembly 数値型を Lean 4 / Mathlib の `BitVec N` のエイリアスとして表現します：

```lean
abbrev I32  := BitVec 32
abbrev I64  := BitVec 64
abbrev F32  := BitVec 32
abbrev F64  := BitVec 64
abbrev V128 := BitVec 128

abbrev Addr32 := BitVec 32
abbrev Addr64 := BitVec 64
```

すべての演算（整数演算、ビット操作、メモリアドレス計算、SIMD レーン操作）は `BitVec N` 上に定義されます。

## 影響

### 肯定的
- **reinterpret はアイデンティティ** — `f32` と `i32` 間の変換はノーオペレーション（無料）
- **Mathlib の `BitVec` 補題を活用** — 豊富な証明済みのビット演算ライブラリ
- **SIMD が自然** — `V128` は `BitVec 128`、レーン抽出はビットスライスで実現
- **演算の統一** — 例えば `wrap_N` は任意の `BitVec N` に対して適用可能
- **型パラメータの柔軟性** — 幅がコンパイル時の自然数であるため、幅に対する汎用的な証明が可能

### 否定的
- **型安全性の低下** — `I32` と `F32` はどちらも `BitVec 32` のため、型レベルで区別されない
- **意味的曖昧性** — 関数が受け取る `BitVec 32` が整数なのか浮動小数点数なのかシグネチャだけでは不明

### 中立的
- `I32` / `F32` のような名前付きエイリアスがドキュメントとしての役割を果たす
- コンテキスト（モジュール名、関数名プレフィックス）が型の意図を十分に伝達

## 検討した代替案

### 算術型ごとの個別型
`structure WasmI32`、`structure WasmF32` など。却下：型間の変換ボイラープレートが膨大であり、reinterpret にラップ/アンラップが必要となり、Mathlib の `BitVec` 補題が直接適用できません。

### Newtype ラッパー
`structure F32 where bits : BitVec 32`。却下：上記と同様の問題に加え、すべてのラッパーに対してインスタンスを個別に定義する必要があります。

### ネイティブ型の使用
Lean の `UInt32`、`UInt64`、`Float` を使用する方式。却下：`UInt32` / `UInt64` は `BitVec` ほど証明をサポートしておらず、`Float` はビット正確ではありません。
