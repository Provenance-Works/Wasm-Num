# ADR-0002: BitVec N を統一表現として採用

| | |
|---|---|
| **ステータス** | 承認済 |
| **日付** | 2025 |
| **決定者** | wasm-num メンテナー |

## コンテキスト

WebAssembly には4つの数値型（i32, i64, f32, f64）と v128 があります。ビットレベルでは、同じ幅の整数と浮動小数点数は交換可能です — `reinterpret` 命令は計算なしで両者を変換します。型システムは以下を扱う必要があります：

- 「任意の32ビット値」に対して機能する演算（reinterpret、メモリ load/store）
- 整数解釈に固有の演算（シフト、div）
- 浮動小数点解釈に固有の演算（fmin、sqrt）
- 任意の型の複数レーンをパッキングする SIMD ベクター

## 決定

すべての WebAssembly 数値型を `BitVec N` のエイリアスとして表現します：

```lean
abbrev I32  := BitVec 32
abbrev I64  := BitVec 64
abbrev F32  := BitVec 32
abbrev F64  := BitVec 64
abbrev V128 := BitVec 128
abbrev Byte := BitVec 8
```

ビットパターンの*解釈*は、型ではなく、どの演算が適用されるかによって決定されます。`reinterpret` は恒等関数です。

## 影響

### 肯定的
- `reinterpretI32AsF32` は文字通り `id` — ゼロコスト、ゼロ証明負担
- メモリ load/store は `BitVec N` 上で統一的に動作 — ラップ/アンラップ不要
- Mathlib の `BitVec` 定理がすべての Wasm 数値型に直接適用可能
- シンプルで小さな型宇宙 — 推論が容易

### 否定的
- 整数操作を浮動小数点意図の値に適用することをコンパイル時に防止できない
- 型シグネチャが I32 と F32 を区別しない（どちらも `BitVec 32`）
- 正しい演算を適用するには規律が必要

### 中立的
- SIMD V128 は `BitVec 128` — レーン抽出は Shape システム（ADR-004）を使用
- 型区別の欠如は仕様の値レベルでの扱いを反映している

## 検討した代替案

### ラッパー型（newtype パターン）
`structure I32 := (val : BitVec 32)` 等を定義する方式。却下：特にメモリ演算と reinterpret で変換ボイラープレートが膨大になります。仕様は値レベルで区別しません。

### Wasm 値の直和型
`inductive WasmValue := | i32 : I32 → ... | f32 : F32 → ...` を定義する方式。却下：あらゆる箇所でパターンマッチングのオーバーヘッドが発生します。各演算が型を静的に知る数値ライブラリには不適切です。

### 型なし Nat/Int
自然数または整数を使用する方式。却下：ビット幅情報が失われ、モジュラー演算のセマンティクスが手動になります。
