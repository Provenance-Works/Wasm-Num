# ADR-0006: 定義と証明の厳格な分離

| | |
|---|---|
| **ステータス** | 承認済 |
| **日付** | 2025 |
| **決定者** | wasm-num メンテナー |

## コンテキスト

Lean 4 では定義と証明を同一ファイルに共存させることが可能です。しかし：

- プルーフチェックはコストが高い（型チェックよりも遅いことが多い）
- 開発中は定義が頻繁に変更される一方、証明は一時的に壊れている場合がある
- 定義のみを必要とするユーザーが証明のコンパイル時間を負担すべきではない
- コードベースには複数のビルドターゲットが必要：定義のみ、定義＋証明、テスト

## 決定

厳格な分離を維持します：

1. **定義ファイル**（`WasmNum/`）には `def`、`structure`、`inductive`、`class`、`abbrev`、`instance` などのみを含め、`theorem`、`lemma`、証明項は含めない
2. **証明ファイル**は `WasmNum/Proofs/` と `Proofs/` に配置し、定義の階層構造をミラーリング
3. **ビルドターゲット**：
   - `WasmNum` — 定義のみ（高速ビルド）
   - `WasmNumProofs` — 定義＋すべての証明
   - `TestAll` — 定義＋テスト

## 影響

### 肯定的
- `lake build WasmNum` が高速 — プルーフチェックなし
- 定義がクリーンで読みやすい（証明の混在なし）
- 証明の失敗が定義の開発をブロックしない
- 明確な所有権：定義ファイル vs. 証明ファイル

### 否定的
- 並行するディレクトリ構造の保守が必要（定義 ↔ 証明）
- 対応する証明を追加せずに定義を追加してしまいやすい
- インポート管理がやや複雑（証明ファイルが定義ファイルをインポート）

### 中立的
- `WasmNumProofs.lean` がすべてを再エクスポート（定義＋証明）
- 一部の証明は `WasmNum/Proofs/`（コロケーションだが別ファイル）に、他はトップレベル `Proofs/` に配置

## 検討した代替案

### コロケーション（同一ファイル）
定理を定義と同一ファイルに配置。却下：開発イテレーションにおけるビルド速度のペナルティが許容できません。

### 証明専用ビルドターゲット
証明を定義ファイルに残しつつ、スキップするビルドフラグを設ける方式。却下：Lean 4 のビルドシステムはこの方法での条件付きコンパイルをサポートしていません。

### 証明用の別パッケージ
証明を完全に別の Lake パッケージに配置。却下：過剰な設計であり、同一パッケージ内の別ファイルで十分です。
