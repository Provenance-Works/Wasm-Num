# ADR-0007: C FFI 不使用 — 純粋 Lean のみ

| | |
|---|---|
| **ステータス** | 承認済 |
| **日付** | 2025 |
| **決定者** | wasm-num メンテナー |

## コンテキスト

Lean 4 はシステムライブラリとの相互運用のために C FFI をサポートしています。wasm-num において C FFI の主な誘惑は以下です：

- ハードウェアによる IEEE 754 浮動小数点演算（`math.h`、Berkeley SoftFloat）
- パフォーマンスクリティカルな操作
- プラットフォーム固有の最適化

しかし、wasm-num の主要な目的は*形式検証* — WebAssembly 数値セマンティクスに対する機械的にチェックされた保証の提供です。

## 決定

wasm-num は C FFI を使用しません。コードベース全体が純粋な Lean 4 です。

- 浮動小数点演算は `WasmFloat` 型クラスで抽象化（[ADR-0001](0001-typeclass-mediated-754-independence.md)）
- テスト用のデフォルトスタブインスタンスが存在
- ハードウェア浮動小数点数を使用したいユーザーは、外部で独自の `WasmFloat` インスタンスを提供する必要あり

## 影響

### 肯定的
- **完全に検証可能** — 信頼されたコンピューティングベースに未検証のコードなし
- **移植性** — ネイティブライブラリ依存なし、Lean がサポートするすべてのプラットフォームでビルド可能
- **再現性** — プラットフォーム依存の動作なし（どこでも同一の結果）
- **監査可能性** — セキュリティと正確性のレビュアーがすべてを検査可能
- **ビルドの簡素化** — C コンパイラやリンカーが不要

### 否定的
- デフォルトの float スタブはプレースホルダ値（カノニカル NaN）を返却 — 実際の浮動小数点計算には使用不可
- ハードウェア浮動小数点数の速度を活用不可
- ユーザーはプロダクション使用時に独自の `WasmFloat` ブリッジを提供する必要あり

### 中立的
- 純粋 Lean はすべての非浮動小数点演算（整数、メモリ、SIMD 構造）に十分
- テストスイートは非浮動小数点パスを検証。浮動小数点依存テストはスタブの分類を使用（これは正確）

## 検討した代替案

### SoftFloat への C FFI
正確な IEEE 754 演算のために Berkeley SoftFloat をリンク。却下：信頼された C コード、ネイティブビルド依存、検証ギャップを導入します。

### Lean 4 ネイティブ Float
Lean 4 の `Float` 型を使用。却下：`Float` はプラットフォーム依存の倍精度であり、ビット正確ではなく、f32 と f64 の両方に対する形式的推論に不適切です。

### オプショナル FFI（ビルドフラグ）
純粋モードと FFI モードの両方をサポート。却下：2つのコードパスの保守が複雑であり、型クラスアプローチ（[ADR-0001](0001-typeclass-mediated-754-independence.md)）がビルドフラグなしでこの柔軟性を既に提供しています。
